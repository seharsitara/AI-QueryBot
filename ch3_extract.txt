2.4.1 Description of Non-Functional Requirement
1. Security
Includes secure authentication, protected API access, hashed password storage,
and controlled access to user-specific documents and chat sessions. API
communication should be deployed over secure transport in production
environments. The system applies practical safeguards for MVP-level privacy and
access protection.
2. Usability
Provides a clean interface, straightforward navigation, and guided workflows for
uploading documents, asking questions, reading citations, and viewing summaries.
The frontend layout is designed to minimize user confusion and support smooth
interaction.
3. Logging and Monitoring
Tracks API/runtime errors, document processing failures, and retrieval/response
behavior through application logging. These logs support debugging, issue
diagnosis, and gradual quality improvement of the platform.
4. Accessibility
Supports readability through clear typography, consistent layout, and
understandable interaction flow. The current implementation emphasizes practical
usability; advanced accessibility features (e.g., speech output or full WCAG
optimization) can be added in future versions.
5. Scalability
Uses vector indexing and retrieval architecture suitable for increasing document
collections and user queries. The design is extensible for future infrastructure
scaling, although distributed load-balancing architecture is outside the current
MVP scope.
6. Performance
Targets an interactive response experience for document retrieval and question
answering under typical usage conditions. Actual latency depends on document
size, retrieval depth, and external LLM API response time; therefore, strict sub-
second guarantees are not assumed.
7. Maintainability
The system follows a modular backend/frontend structure that supports easier
feature enhancement, bug fixing, and configuration updates. This architecture
allows future improvements such as model replacement, retrieval tuning, and
interface refinement with limited disruption.

3.1 Introduction
The analysis phase translates project requirements into clear system behavior by
identifying user goals, system boundaries, interaction flows, and processing logic. For
QueryBot, analysis focuses on how a registered user interacts with a document-centric
AI assistant to upload files, generate embeddings, ask context-aware questions, receive
source-grounded responses, and maintain persistent chat sessions.
QueryBot is implemented as a web-based client-server system where the frontend
(Next.js) manages user interaction and the backend (FastAPI) handles authentication,
document processing, retrieval-augmented generation (RAG), citation mapping, and
chat/session persistence. Since the system does not include an administrative role, all
analysis artifacts in this chapter model only the User as the primary actor and the
System as the supporting actor that validates requests, executes processing pipelines,
and returns results.

4.1 Architecture Diagram
4.1.1 Introduction
QueryBot is designed using a three-tier architecture that separates presentation,
application logic, and data management. This structure improves maintainability,
supports future scalability, and keeps responsibilities clearly divided across system
components.
∩ü¼ Presentation Tier: User-facing web interface for authentication, document
handling, and conversational interaction.
∩ü¼ Application Tier: Backend services that execute authentication, document
processing, retrieval, and response generation workflows.
∩ü¼ Data Tier: Persistent and semantic storage for transactional records and
vectorized document knowledge.
4.1.2 Component Breakdown
1. Frontend (Presentation Layer)
i. Technology: Next.js (React) with Tailwind CSS
ii. Responsibilities: User registration/login screens, document upload workflows,
chat interface, citation visibility, summary and export actions.
2. Backend API (Application Layer)
i. Technology: FastAPI (Python)
ii. Responsibilities: API endpoint handling, JWT-based authorization, document
parsing and preprocessing, chunking/indexing pipeline, retrieval orchestration,
LLM request handling, citation mapping, and chat/session management.
3. Data Layer (Persistence + Retrieval)
i. SQLite: Stores user records, authentication-linked entities, sessions, chat
messages, and operational metadata.
ii. ChromaDB: Stores embeddings and chunk metadata for semantic retrieval and
relevance matching.
4. AI Services
i. LangChain: Coordinates retrieval and prompt orchestration within the RAG
pipeline.
ii. Groq (LLM endpoint): Produces context-aware answers and summaries from
retrieved evidence.
