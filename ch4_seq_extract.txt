4.3 Data Flow Diagrams
4.3.1 Data Flow diagram (Level 0)
The Context Diagram serves as the foundational view of the QueryBot system's data
architecture. It establishes the system boundary by representing the entire
applicationΓÇöincluding the Next.js frontend and FastAPI backendΓÇöas a single high-
level process, identified as Process 0.0. This abstraction is essential for defining how
the system interacts with the outside world before detailing internal data handlings.
External Entities and Interactions:
1. User Interaction: The User acts as the primary external entity. Data flowing
from the User into the system includes authentication credentials for secure
access, document files (such as PDF and DOCX) for processing, and natural
language queries intended for the AI. Conversely, the system provides the User
with essential data outputs, including login/registration status, document upload
confirmations, and the final AI-generated responses, summaries, and chat history.
2. LLM Service Integration: Since the core AI intelligence is handled externally
via Groq, the LLM Service is treated as a secondary external entity. The
QueryBot system sends "Augmented Prompts"ΓÇöwhich are refined versions of
the user's query combined with relevant document contextΓÇöto the LLM. The
service then processes this data and returns generated text responses back to the
system for final delivery to the user.
Data Flow Principles:
3. System Boundary: At this level, internal data persistence layers like
PostgreSQL (for structured data) and ChromaDB (for vector embeddings) are
kept within the system boundary and are not illustrated. This ensures that the
Context Diagram remains focused strictly on external interfaces and data
exchange.
4. Logical Processing: The diagram illustrates that every user request undergoes a
transformation within the QueryBot boundary, where external knowledge (from
documents) and external intelligence (from the LLM) are synthesized to produce
the final output.

4.3.2 Data Flow Diagram (Level 1)
Functional Decomposition (QueryBot)DFD Level 1 decomposes Process 0.0
(QueryBot System) into major internal processes to show how data is validated,
transformed, stored, retrieved, and returned to the user. At this level, the system
behavior is represented through five core processes:
1. User Authentication:
This process handles registration and login requests received from the user.
Credentials are validated against stored account data in SQLite, and an
authentication result is returned. This step ensures that document and chat
operations are performed only within the authorized user scope.
2. Document Upload and Processing:
This process receives uploaded files (PDF, DOCX, and TXT), validates the file
format and size, extracts text, and prepares the content for retrieval. After
preprocessing, document metadata and extracted text are stored in SQLite, while
vector representations are indexed in ChromaDB for semantic search.
3. Semantic Retrieval:
It receives natural-language questions from the user and retrieves relevant context
from ChromaDB. The process transforms the user query into a retrieval-ready
form and gathers the most relevant document chunks needed for grounded
answer generation.
4. AI Response Generation:
This process takes the retrieved context and constructs an augmented prompt.
This prompt is sent to the external LLM Service (OpenRouter via LangChain),
which returns generated text. QueryBot then structures the returned content into a
response-ready output for the application.
5. Chat History and Output Management:
This final process stores user queries and assistant responses in SQLite as
persistent session history. It also returns the final outputs to the user, including
contextual answers, summaries, and conversation history views.
Summary: In summary, DFD Level 1 shows that QueryBot processes user input
through a controlled pipeline:Authenticate $\rightarrow$ Process Document
$\rightarrow$ Retrieve Context $\rightarrow$ Generate AI Response
$\rightarrow$ Persist and Deliver Output.This decomposition accurately reflects the
implemented frontend-backend workflow and the role of SQLite, ChromaDB, and
external LLM integration in the project.
